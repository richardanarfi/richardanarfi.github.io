Why exact-match metrics fail for LLM systems

Alternate-valid answers, rubric-based evaluation, and why exact match systematically under-measures real LLM capability.

The appeal of exact match

Exact match is the simplest evaluation metric: does the model's output exactly equal the expected answer? It's deterministic, cheap to compute, and easy to explain. For tasks with a single correct answer (what is 2+2?), it works perfectly. The problem is that most interesting LLM tasks don't have a single correct answer.

Where it breaks down

Consider a simple question: "Who is the CEO of Anthropic?" The golden answer in your test set says "Dario Amodei." The model outputs "Dario Amodei is the CEO of Anthropic." Exact match: fail. The model gave a correct, complete answer, but because it included extra context, it gets zero credit.

This isn't a contrived example. It happens constantly in production evaluation. A model generates a SQL query that returns the same rows but uses a column alias instead of the raw column name. A model writes a summary that covers all the key points but reorders them. A model answers a math problem correctly but shows its work in a different format. In all these cases, exact match says "wrong" when the output is correct.

The damage compounds at scale. If 15-20% of correct outputs are marked as failures due to surface-level differences, your metrics are systematically pessimistic. You can't trust your pass rate. You can't tell whether a prompt change actually helped or whether the delta is just noise from formatting variation. Engineers waste time triaging "regressions" that are actually correct outputs in a different form.

Alternate-valid answers

The first fix is to support multiple valid answers per test case. Instead of one golden string, store an array of acceptable outputs. When the model produces a new correct variant, add it to the list. Over time, the test set accumulates valid alternatives and the false-failure rate drops.

This is straightforward to implement but hard to maintain. Somebody has to decide which alternatives are valid. In my experience building evaluation pipelines, the initial test set has one alternative per case, and after a few months of iteration you end up with 3-5 alternatives per case. The maintenance burden grows linearly with the number of test cases and the number of prompt iterations. You need a system for this, not a manual process.

Structured assertions

A better approach for structured outputs (SQL, JSON, API calls) is to evaluate the output's structure rather than its string representation. Two SQL queries are equivalent if they return the same result set, regardless of column ordering, aliasing, or whitespace. Two JSON objects are equivalent if they have the same keys and values, regardless of key ordering.

This requires a decomposition step: parse the output into its structural components, then compare each component semantically. For a SQL query, that means comparing the WHERE clause conditions (do they filter the same rows?), the SELECT columns (cosmetic difference, doesn't affect results), and the JOIN structure (changes the result set). A difference in SELECT columns is cosmetic. A difference in WHERE conditions is semantic. This distinction matters because it's the difference between a false alarm and a real regression.

Rubric-based judging

For open-ended outputs (summaries, explanations, conversational responses), neither exact match nor structural comparison works. The output space is too large and too varied. Here, rubric-based evaluation with an LLM judge is the practical answer.

A rubric defines 3-5 evaluation dimensions with explicit criteria for each score level. For a summary task, the dimensions might be: faithfulness (are all claims supported by the source?), completeness (are all key points covered?), coherence (does it read well?), and conciseness (is it appropriately brief?). Each dimension gets a 1-5 score with concrete anchors for what each level means.

The critical implementation detail is to use separate judges for separate dimensions. A single judge scoring all dimensions at once exhibits anchor bias, where a low score on one dimension drags down scores on others. Running independent judges per dimension eliminates this and produces more reliable scores.

Using multiple judge models (e.g., running the same rubric through three different models) adds another layer of reliability. When all three judges agree, you can trust the score. When they disagree, you have a signal that the case is ambiguous and may need human review. In practice, three-judge panels with debate resolution for disagreements reach 96-97% agreement with human annotators, compared to 89% for a single judge.

When to use what

The right evaluation method depends on the task:

  • Exact match works for tasks with truly unique answers: math with integer results, classification labels, boolean yes/no. Use it where it fits because it's fast and deterministic.
  • Alternate-valid with structural comparison works for structured outputs (SQL, API calls, JSON, tool call sequences) where equivalence can be defined programmatically.
  • Rubric-based LLM judging is necessary for open-ended generation where the output space is too large to enumerate.

The mistake most teams make is using exact match for everything because it's easy to set up, then wondering why their metrics are noisy and their engineers spend half their time triaging false failures. The investment in proper evaluation infrastructure pays for itself quickly: less wasted triage time, more trustworthy metrics, and faster iteration cycles.
