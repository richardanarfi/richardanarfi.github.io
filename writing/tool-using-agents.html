How to evaluate tool-using agents beyond pass/fail

Tool choice, recovery behavior, hallucination rates, and why agent evaluation should separate process from outcome.

The pass/fail trap

Most agent benchmarks work like this: give the agent a task, check if the final answer is correct, report a pass rate. This tells you almost nothing useful. Two agents can both score 70% and be failing in completely different ways. One might be picking the wrong tools. The other might be picking the right tools with wrong parameters. A third might be doing everything right but failing to recover when a tool call returns an error. Pass/fail treats all of these as the same kind of failure, which makes it useless for debugging or improvement.

The deeper problem is that pass/fail can't distinguish a good process from a lucky outcome. An agent can stumble into the right answer through the wrong tools. It can brute-force a calculation by dumping everything into a code interpreter when a targeted API call would have been faster and more reliable. If your metric only checks the final answer, these look identical. But in production, the agent that uses the right tools is faster, cheaper, and more reliable than the one that gets lucky.

Separate process from outcome

The fix is to evaluate the agent's decision process independently from the final result. This means logging and scoring every intermediate step: which tool was selected, what parameters were passed, whether the tool call succeeded, and what the agent did with the result.

Concretely, for each task you evaluate at least three dimensions:

  • Tool selection accuracy: Did the agent invoke the right tool for the task? For multi-step tasks, did it invoke the right sequence of tools? This extends naturally to precision and recall over the expected tool set.
  • Parameter correctness: Given the right tool, were the arguments correct? A web search with the right query is very different from a web search with a vague or fabricated query. Scoring parameters separately from tool selection pinpoints whether the problem is routing (picking the wrong tool) or grounding (picking the right tool but with bad inputs).
  • Task success: Did the final answer match the expected output or an acceptable alternative? This is still important, but it's one signal among several, not the only one.

With all three dimensions, a failure analysis becomes actionable. "Tool selection dropped 8% on retrieval tasks after adding the calculator tool" is something you can investigate and fix. "Pass rate dropped 3%" is not.

Recovery behavior

One of the most informative dimensions is what the agent does when something goes wrong. Tools fail. APIs return errors. Search results come back empty. The question is: does the agent handle this gracefully?

In practice, I've observed four recovery patterns:

  1. Retry with correction: The agent recognizes the error, adjusts the parameters, and retries. This is the best outcome.
  2. Fallback to alternative: The agent switches to a different tool that can accomplish the same goal. Also good, though it requires the agent to understand tool equivalences.
  3. Blind retry: The agent repeats the exact same failing call, sometimes multiple times. This wastes tokens and time while accomplishing nothing.
  4. Abandonment: The agent gives up and either returns an incomplete answer or says it can't help. This is the worst outcome for the user.

Recovery score has the highest variance across models of any metric I've measured. Models that look similar on pass/fail diverge sharply on recovery. This makes sense: recovery requires the model to interpret an error message, reason about what went wrong, and generate a corrective action. It's a harder capability than the initial tool selection, and it's where model quality shows up most clearly.

Tool interaction effects

A subtle failure mode that pass/fail completely misses is tool interaction. Adding a new tool to an agent's toolkit can degrade performance on existing tasks, even though the new tool works fine on its own. This happens because tool descriptions compete for attention in the prompt. A new tool with vocabulary that overlaps an existing tool (e.g., both have "search" in the description) can confuse routing and cause the agent to invoke the wrong one.

To catch this, you need a regression framework: run the same task set before and after adding the new tool, and measure the delta. The key insight is that you should measure this per existing tool, per quality dimension. An aggregate "pass rate dropped 2%" hides the fact that one specific tool's selection accuracy collapsed by 15% while everything else was fine.

Interaction effects can also be non-linear. Two tools might each cause a small individual regression, but when both are added together the combined effect is much worse than the sum of individual effects. Game-theoretic methods like the Shapley Interaction Index can quantify this: the portion of the combined degradation that can't be predicted from individual tool effects. If the interaction index is non-zero, you have an emergent conflict that individual testing would never reveal.

LLM-as-judge for qualitative dimensions

Some evaluation dimensions resist programmatic scoring. When multiple tools contributed to a response, is the result a coherent synthesis or a disjointed patchwork? When a new tool is added, does its information leak into unrelated responses? These are judgment calls that require understanding context and intent.

LLM judges can evaluate these dimensions, but the implementation matters. The most important design choice is to use separate judges for separate dimensions. A single judge scoring "synthesis quality" and "capability bleed" and "routing justification" simultaneously will exhibit cross-dimension anchor bias. A low score on synthesis quality unconsciously pulls down the routing justification score, even if routing was fine. Independent judges eliminate this.

The second important choice is using multiple models as judges. A three-judge panel (e.g., different model sizes or families) provides a built-in reliability signal. Unanimous agreement means you can trust the score. A 2-1 split means the case is worth a closer look. Running a structured debate where a fourth judge reviews the disagreeing judges' reasoning and renders a final verdict pushes agreement with human annotators from 94% (majority vote) to 97% (debate resolution).

What a diagnostic benchmark looks like

Putting this together, a diagnostic agent benchmark produces a report that looks more like a test suite than a single number:

  • Tool selection accuracy per tool, per task category
  • Parameter accuracy per tool
  • Recovery success rate and recovery pattern distribution
  • Task success rate (the traditional pass/fail, but now contextualized)
  • Regression scores when tool configurations change
  • Interaction indices for tool pairs
  • Judge scores on qualitative dimensions

This is more work to build and maintain than a simple pass/fail benchmark. But the payoff is that every metric is actionable. When something drops, you know where to look. When you make a change, you can measure its effect on the specific dimension you were targeting instead of hoping the aggregate pass rate moves in the right direction. Evaluation should help you build better agents, not just tell you whether you got lucky.
