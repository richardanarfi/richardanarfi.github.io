GRPO Arithmetic

An exploration of Group Relative Policy Optimization (GRPO) for teaching arithmetic to a small language model, investigating how reward design, group size, and training dynamics affect both accuracy and output quality.

Motivation

Why This Experiment

GRPO vs PPO for capability elicitation

PPO requires training a separate value network and reward model, adding complexity and compute. GRPO sidesteps this by generating multiple completions per prompt and using the relative ranking within each group as the baseline. I wanted to understand how this trade-off plays out in practice: does GRPO's simplicity come at the cost of sample efficiency or stability?

Reward hacking as an alignment microcosm

Arithmetic is a clean testbed for reward design problems. A naive reward (just correctness) causes the model to collapse to bare numeric output, technically correct but stripping all natural language. This is reward hacking at small scale, and the fix (multi-signal reward shaping) mirrors the same challenges in larger alignment work.

How GRPO Works

For each training step, GRPO samples a batch of prompts and generates multiple completions per prompt (the "group"). Each completion is scored by the reward function. The policy gradient is computed using the relative advantage within each group: completions that score higher than the group mean get reinforced, and lower ones get suppressed.

This is simpler than PPO because there's no separate value network or reward model to train. The group comparison acts as a built-in baseline, reducing variance without additional parameters.

from trl import GRPOConfig, GRPOTrainer

    args          = train_args,
    train_dataset = train_dataset,            # 10,000 arithmetic problems
)

trainer.train()  # GRPO handles generation, scoring, and updates internally

Two-Signal Reward Function

The reward is a weighted combination of two signals: correctness (did the model get the right answer?) and natural language (did it respond with words, not just a bare number?). The second signal prevents reward hacking. Without it, the model learns to strip all text and output only digits.

Python 
def reward_correctness(prompts, completions, answer, **kwargs):
    """1.0 for correct answer, 0.0 otherwise.
    Extracts the last number in the completion as the predicted answer."""
    rewards = []
    for completion, target in zip(completions, answer):
        content = completion[-1]["content"]
        matches = re.findall(r"([+-]?\d+(?:\.\d*)?)", content)
        predicted = int(float(matches[-1])) if matches else None
        rewards.append(1.0 if predicted == target else 0.0)
    return rewards

def reward_has_words(prompts, completions, **kwargs):
    """1.0 if completion contains natural language, 0.0 if just numbers."""
    return [
        1.0 if re.search(r'[a-zA-Z]{2,}', c[-1]["content"]) else 0.0
        for c in completions
    ]

# Weighted combination: correctness matters more, but words are rewarded too
# reward_weights = [0.75, 0.5]

The composite scoring function reflects both signals: a correct answer with natural language gets full credit, correct without words gets half credit, and incorrect gets zero. This forces the optimization to balance accuracy with output quality.

Key Training Parameters

The batch configuration in GRPO is tricky because it controls how many prompts are sampled, how many completions are generated per prompt (the group size), and how many sequences contribute to each optimizer step.

Python 
train_args = GRPOConfig(
    # GRPO & Rollouts
    temperature           = 0.75,   # Slightly greedy generation
    top_p                 = 0.9,
    max_completion_length = 48,     # Arithmetic answers are short

    # Batch: 2 prompts x 8 completions = 16 sequences per step
    per_device_train_batch_size = 16,
    num_generations             = 8,  # GRPO group size

    # Optimization
    learning_rate  = 5e-6,
    max_steps      = 300,
    warmup_steps   = 10,
    weight_decay   = 0.01,

    # Reward
    reward_weights = [0.75, 0.5],   # [correctness, has_words]
    scale_rewards  = True,          # Normalize rewards within group

    # vLLM for fast generation
    use_vllm = True,
)

What I Tuned

Through iterative experimentation, the most impactful changes were: doubling the GRPO group size from 4 to 8 (more diverse completions per prompt give a better estimate of relative advantage), enabling reward scaling (normalizing within each group), rebalancing reward weights to increase the natural language signal, and tuning the learning rate from 1e-6 to 5e-6 for faster convergence within the memory constraints of a T4 GPU.

Evaluation Strategy

The model is evaluated every 20 training steps on a 200-example validation set using greedy decoding (temperature=0). The best checkpoint is selected by the composite score (penalizing correct-but-no-words answers), not raw accuracy. This ensures the model doesn't sacrifice natural language for marginal accuracy gains.

Final testing uses two held-out sets: in-domain (400 problems from the same distribution as training) and out-of-domain (400 problems with different number ranges or operations). The test score is the average of both, testing generalization beyond the training distribution.

Python 
class GreedyEvalCallback(TrainerCallback):
    """Periodically runs greedy decoding on validation set,
    saves best checkpoint by composite score."""

    def on_step_end(self, args, state, control, **kwargs):
        if state.global_step % self.eval_every != 0:
            return

        result = evaluate_model_vllm(
            self.vllm, self.tokenizer, self.val_dataset
        )

        if result.score > self.best_score:
            self.best_score = result.score
            self.best_step = state.global_step
            trainer.save_model(self.best_ckpt_dir)
            print(f"New best: {result.score:.1f}% at step {state.global_step}")

The generalization gap

In-domain accuracy (66.8%) was noticeably higher than out-of-domain (57.0%), showing that while GRPO teaches the model how to do arithmetic, generalization to unseen number ranges remains the harder challenge. The gap suggests the model partially memorizes patterns rather than learning fully general arithmetic reasoning, an open question for RL-trained capabilities more broadly.

Reward shaping worked

The dual reward signal maintained natural language responses throughout training with zero penalty. This shows that multi-signal reward shaping (weighting has_words at 0.5) effectively prevents the model from collapsing to bare numeric output, a small-scale demonstration of the same reward design principles used in larger alignment work.

Insights

Lessons Learned

Doubling the group size from 4 to 8 completions per prompt was the single biggest improvement across iterations. A larger group gives GRPO a better estimate of the relative advantage — with only 4 samples, the ranking is noisy and the gradient signal is unreliable. With 8 samples, the model gets a clearer signal about which response patterns are better. This mirrors findings in the GRPO literature on the bias-variance trade-off in group size selection.

Without the has_words reward, the model quickly learns to output bare numbers, maximizing correctness at the expense of natural language. The weighted combination (0.75 correctness + 0.5 has_words) creates a reward landscape where the model is incentivized to be both correct and articulate. If reward hacking appears this readily in basic arithmetic, it's clear why reward design for more complex capabilities requires such careful thought.

The bottleneck in GRPO training is generation: producing 8 completions per prompt for every training step. Without vLLM's optimized inference (PagedAttention, continuous batching), the generation phase would dominate wall time and make iteration impractical. With vLLM, the full training loop completes in under 10 minutes on a T4, enabling rapid experimentation with reward weights, group sizes, and learning rates.

The best checkpoint was at step 260, not the final step 300. Training beyond the optimum caused slight degradation, a common pattern in RL where the policy overshoots. Validating every 20 steps and saving the best by composite score (not raw accuracy) ensured the peak was captured while respecting the natural language constraint.
