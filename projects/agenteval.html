ToolEval

Most agent benchmarks measure final-answer accuracy and stop there. ToolEval evaluates the process: did the agent pick the right tool, pass correct parameters, compose calls in the right order, and recover gracefully when things go wrong?

Motivation

The Problem

Pass/fail misses process quality

An agent can arrive at the correct answer through the wrong tools, like using a web search when a database query was appropriate, or brute-forcing a calculation with code when a calculator tool was available. Task success alone doesn't distinguish efficient tool use from lucky outcomes.

New tools cause silent regressions

Adding a new tool to an agent's toolkit can degrade its performance on existing tasks. The new tool description competes for attention in the prompt, confuses routing, or "starves" existing tools of invocations. Without a regression framework, these effects go undetected until production.

Architecture

Three-Step Evaluation Pipeline

Every evaluation runs the same three steps (baseline, perturbation, and forward check) regardless of which tools are being tested.

Baseline → Perturbation → Forward Check

1. Baseline. Run the full task set with the current tool configuration. Every tool call is traced: which tool was selected, what parameters were passed, whether it succeeded, and the final task outcome. This establishes ground truth performance.

2. Perturbation (Reverse Check). Add a new tool (the Tool Under Test) to the agent's toolkit and re-run the same tasks. These tasks don't require the new tool; they belong to existing tools. Any performance delta indicates regression caused by the new tool's presence in the prompt.

3. Forward Check. Run the new tool's own task set with all tools active, verifying it functions correctly alongside existing tools. Since there is no "before" baseline for new-tool tasks, this step uses absolute pass rates.

Python 
class EvalPipeline:
    def run(self, agent, tools_baseline, tool_under_test, tasks):
        # Step 1: Baseline — existing tools only
        baseline_traces = self.execute(agent, tools_baseline, tasks.existing)

        # Step 2: Perturbation — add new tool, re-run same tasks
        tools_combined = tools_baseline + [tool_under_test]
        perturbed_traces = self.execute(agent, tools_combined, tasks.existing)

        # Step 3: Forward — new tool's tasks with all tools
        forward_traces = self.execute(agent, tools_combined, tasks.new_tool)

        return EvalReport(
            regression=self.score_regression(baseline_traces, perturbed_traces),
            forward=self.score_forward(forward_traces),
            interaction=self.score_interaction(baseline_traces, perturbed_traces),
        )

Task Schema

Each benchmark task specifies a user query, expected tool calls (with parameters), a gold answer, and metadata for difficulty and category. Tasks span four categories: retrieval (find information), calculation (compute results), multi-step (chain 2-3 tools), and recovery (handle tool failures).

The acceptable_alternatives field captures alternate-valid tool sequences. An agent that solves the task with a single code_interpreter call instead of three separate tool calls is still correct, and ToolEval scores both paths.

Metric Suite

All metrics are computed per tool, per quality dimension, never averaged across tools. This granularity pinpoints exactly which tool was hurt on which dimension.

Core Metrics

  • Tool Selection Accuracy (TSA): Binary per task. Did the agent invoke the correct tool? For multi-tool tasks, extends to precision/recall/F1 over the expected tool set.
  • Parameter Accuracy: Per tool call. Were the arguments correct? Scored as exact match for categorical params, tolerance-based for numeric params.
  • Task Success Rate: Binary. Did the final answer match the gold or an acceptable alternative?

Interaction Metrics

  • Tool Regression Score (TRS): (TSA_perturbed - TSA_baseline) / TSA_baseline. Negative = regression. Paired t-test for significance.
  • Tool Dominance Index (TDI): Measures invocation rate drop for existing tools after adding the new tool. TDI > 0 = starvation.
  • Routing Entropy: Shannon entropy of the tool invocation distribution. Delta reveals whether routing is destabilized.
  • Shapley Interaction Index: Non-linear interaction between tool pairs, capturing the portion of the combined effect not predicted from individual tool effects.
Python 
def tool_regression_score(baseline_traces, perturbed_traces, tool_name):
    """Compute TRS for a specific tool across its expected tasks."""
    baseline_tsa = np.mean([
        t.tool_correct for t in baseline_traces
        if t.expected_tool == tool_name
    ])
    perturbed_tsa = np.mean([
        t.tool_correct for t in perturbed_traces
        if t.expected_tool == tool_name
    ])
    trs = (perturbed_tsa - baseline_tsa) / baseline_tsa

    # Statistical significance via paired t-test
    _, p_value = stats.ttest_rel(
        [t.tool_correct for t in baseline_traces if t.expected_tool == tool_name],
        [t.tool_correct for t in perturbed_traces if t.expected_tool == tool_name],
    )
    return TRS(value=trs, p_value=p_value, significant=p_value < 0.05)

Failure Taxonomy

ToolEval classifies every incorrect tool call into a failure type. This taxonomy drives actionable feedback. "The agent hallucinates tools" is more useful than "accuracy dropped 3%."

  • Tool Hallucination: Agent invokes a tool that doesn't exist in the toolkit. Often caused by model training on broader tool sets.
  • Parameter Fabrication: Correct tool, but arguments contain invented values not derivable from the context.
  • Wrong Tool Selection: Agent picks a real but incorrect tool (e.g., web_search instead of database_query).
  • Composition Error: Correct individual tools, but wrong ordering or missing intermediate steps.
  • Premature Termination: Agent stops before completing all required tool calls.
  • Recovery Failure: Tool returns an error, and the agent gives up instead of retrying or using an alternative.
Python 
def classify_failure(trace, expected):
    """Classify a failed tool call into the failure taxonomy."""
    if trace.tool_name not in available_tools:
        return FailureType.TOOL_HALLUCINATION

    if trace.tool_name != expected.tool:
        return FailureType.WRONG_TOOL

    if not params_match(trace.params, expected.params):
        return FailureType.PARAM_FABRICATION

    if trace.order != expected.order:
        return FailureType.COMPOSITION_ERROR

    if trace.status == "error" and not trace.retry_attempted:
        return FailureType.RECOVERY_FAILURE

    return FailureType.PREMATURE_TERMINATION

Regression is model-dependent

Claude 3.5 showed the smallest TRS (-2.1%) when adding new tools, suggesting more robust routing. GPT-4 regressed more on multi-step tasks (-9.4%) where the new tool description competed with existing tool descriptions for context.

Shapley reveals hidden conflicts

The Shapley Interaction Index between web_search and database_query was -0.45, indicating strong negative interaction. Combining them degraded performance more than either alone. Individual tool metrics missed this because both tools performed well in isolation.

Insights

Lessons Learned

In 23% of "passing" tasks, the agent used a suboptimal tool path, reaching the correct answer through unnecessary tool calls or by ignoring the most appropriate tool. Outcome-only metrics would mark these as successes, hiding systemic routing issues that degrade latency and reliability at scale.

Rephrasing a single tool description changed tool selection accuracy by up to 18%. Tools with overlapping vocabulary (e.g., "search" in both web_search and database_query) caused the most confusion. The framework now includes a description similarity check that flags potential conflicts before evaluation.

When a tool call fails, the best agents retry with corrected parameters or fall back to an alternative tool. The worst agents either repeat the exact same failing call or abandon the task entirely. Recovery score had the highest variance across models of any metric, and it's where model capabilities diverge most sharply.

Two tools can each perform well alone but degrade the agent when combined, an emergent interaction effect. The Shapley Interaction Index quantifies this directly. In one case, adding both calculator and code_interpreter caused a -0.55 SII on task success, even though each tool alone caused only -0.1. Without interaction metrics, you'd never predict the combined degradation.
