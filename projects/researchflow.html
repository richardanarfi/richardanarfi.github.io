PromptLab

Prompts degrade silently. Model updates, new features, and context changes erode quality over time. PromptLab gives prompts the same regression testing, failure triage, and versioned ground truth that software gets from unit tests and CI.

Motivation

The Problem

Prompts are untested code

Most prompt changes are validated by eyeballing a few examples. There's no regression suite, no diff view, no CI gate. A well-intentioned edit to improve one behavior silently breaks three others, and nobody notices until users complain.

Ground truth becomes stale

Golden test sets are written once and rarely updated. As the model improves, it produces outputs that are correct but different from the golden, failing the regression test. Engineers waste time triaging "false regressions" that are actually improvements. Without a system for evolving ground truth, test suites become a burden instead of a safety net.

Regression Runner

The runner executes a prompt version against a golden test set. Each test case has an input, one or more expected outputs (the ground truth supports multiple valid alternatives), and metadata. The runner compares the actual output to all alternatives and flags mismatches as failures.

Python 
class RegressionRunner:
    def run(self, prompt_version, test_suite):
        results = []
        for case in test_suite.cases:
            actual = self.execute_prompt(prompt_version, case.input)
            match = self.compare(actual, case.expected_alternatives)

            results.append(TestResult(
                case_id=case.id,
                input=case.input,
                expected=case.expected_alternatives,
                actual=actual,
                passed=match.is_match,
                matched_alternative=match.alt_index,
            ))

        passed = sum(1 for r in results if r.passed)
        return RegressionReport(
            prompt_version=prompt_version.id,
            total=len(results),
            passed=passed,
            failed=len(results) - passed,
            pass_rate=passed / len(results),
            failures=[r for r in results if not r.passed],
        )

Prompt versions are tracked with structured metadata: version number, author, change description, timestamp, and parent version. This creates an audit trail so you can always trace back to which prompt change caused a regression.

Structural Decomposition

For each failure, PromptLab decomposes the expected and actual outputs into structured components and classifies each difference as semantic (changes the meaning/result) or cosmetic (same meaning, different surface form).

This is critical because many "failures" are actually cosmetic. The model produced a correct output that's formatted differently, uses different phrasing, or orders items differently. Without structural decomposition, every cosmetic difference triggers a false alarm.

Python 
def decompose_diff(expected, actual, schema):
    """Decompose output differences into semantic vs cosmetic."""
    diffs = []

    for field in schema.fields:
        exp_val = extract_field(expected, field)
        act_val = extract_field(actual, field)

        if exp_val == act_val:
            continue

        diff_type = classify_diff(field, exp_val, act_val)
        diffs.append(StructuralDiff(
            field=field.name,
            expected=exp_val,
            actual=act_val,
            diff_type=diff_type,  # SEMANTIC or COSMETIC
            impact=field.impact,  # Does this field affect results?
        ))

    return DecompReport(
        diffs=diffs,
        has_semantic_diff=any(d.diff_type == "SEMANTIC" for d in diffs),
        cosmetic_only=all(d.diff_type == "COSMETIC" for d in diffs),
    )

Three-Judge Panel

Failures that aren't resolved by structural decomposition go to a three-judge panel. Each judge independently evaluates: "Does the actual output produce the same result as the expected output?" Using three different models (opus, sonnet, haiku) provides reasoning diversity.

Evaluation Protocol

  1. Each judge receives: the input, the expected output, the actual output, and the structural decomposition
  2. Each judge independently renders a verdict: PASS (semantically equivalent) or FAIL (different result)
  3. Unanimous agreement → accept immediately
  4. 2/1 split → spawn a debate arbiter (claude-opus) that reviews all three judges' reasoning and renders a final verdict
  5. Fallback: if arbiter fails, majority vote (2 of 3 wins)
Python 
async def judge_panel(failure, decomposition):
    """Three-judge evaluation with debate resolution."""
    judges = [
        Judge(model="claude-opus", style="strict"),
        Judge(model="claude-sonnet", style="balanced"),
        Judge(model="claude-haiku", style="efficient"),
    ]

    verdicts = await asyncio.gather(*[
        j.evaluate(failure, decomposition) for j in judges
    ])

    # Check for unanimous agreement
    if all(v.verdict == verdicts[0].verdict for v in verdicts):
        return Consensus(
            verdict=verdicts[0].verdict,
            resolution="unanimous",
            confidence="high",
        )

    # Disagreement → debate arbiter
    arbiter = Judge(model="claude-opus", role="arbiter")
    final = await arbiter.resolve_disagreement(
        failure, decomposition, verdicts
    )
    return Consensus(
        verdict=final.verdict,
        resolution="debate",
        reasoning=final.reasoning,
    )

Ground Truth Evolution

When the judge panel determines that a "failing" output is actually correct (semantically equivalent despite surface differences), PromptLab automatically adds the new output as an alternative to the ground truth. Existing alternatives are never removed; only new ones are added.

This means the test suite gets more robust over time: it accumulates valid variations, reducing false regressions on future runs while preserving all previously accepted outputs as valid baselines.

Python 
def update_ground_truth(original_gt, judge_results):
    """Add judge-approved alternatives to ground truth.
    Never removes existing alternatives — only appends."""
    updated = deepcopy(original_gt)

    for result in judge_results:
        if result.verdict != "PASS":
            continue  # Only add alternatives for PASS cases

        case = updated.find_case(result.case_id)
        new_alt = result.actual_output

        # Add as new alternative (preserving all existing)
        case.expected_alternatives.append(new_alt)
        case.annotations.append({
            "added_by": "judge_panel",
            "resolution": result.resolution,
            "reasoning": result.reasoning,
            "timestamp": datetime.utcnow().isoformat(),
        })

    return updated

Verification

Before writing the updated ground truth, PromptLab runs five checks:

  1. Entry count preserved (no cases added or removed)
  2. No existing alternatives removed
  3. All case IDs preserved
  4. New alternatives come from actual LLM outputs (not fabricated)
  5. Tally consistency: judge_passed + judge_failed = total failures

Structural decomposition catches most cosmetic diffs

62% of false regressions were caught by structural decomposition alone, before the judge panel was needed. These were formatting differences, field reordering, and synonym substitutions that a simple string comparison would flag as failures.

After 12 prompt iterations, the false regression rate dropped from 38% (iteration 1) to 12% (iteration 12) as the ground truth accumulated valid alternatives. The test suite became a better safety net with each run, catching real regressions without crying wolf on cosmetic changes.

Walkthrough

Example: A False Regression

Tracing a single test case through the full pipeline to show how PromptLab prevents wasted triage time.

Input: "Show me all orders placed last Tuesday that included express shipping"
Expected: SELECT order_id, customer, total FROM orders WHERE order_date = '2026-03-17' AND shipping_method = 'express'
Actual: SELECT order_id, customer, total, status FROM orders WHERE order_date = '2026-03-17' AND ship_type = 'express'
Status: FAIL (string mismatch)

SELECT clause: Extra column "status" → COSMETIC (selected columns don't affect which rows are returned)
WHERE column: "shipping_method" vs "ship_type" → SEMANTIC (different column name, need to check if they reference the same data)
WHERE value: Identical → no diff
Verdict: Has semantic diff → escalate to judge panel

Judge A (opus): PASS. "shipping_method and ship_type are aliases for the same column in this schema. Same result set."
Judge B (sonnet): PASS. "SELECT clause difference is cosmetic. WHERE column is a known alias in this database schema."
Judge C (haiku): PASS. "Equivalent query, cosmetic differences only."
Resolution: Unanimous PASS

The actual query is added as a second valid alternative for this test case. Future runs will accept either form without flagging a regression. The annotation records which judges approved it and why.

Insights

Lessons Learned

Before PromptLab, engineers spent 40% of their prompt iteration time investigating failures that turned out to be cosmetic or semantically equivalent. Automating this triage freed up time for actual prompt improvement, and the team shipped 3x more prompt iterations in the same time period.

Early versions of the pipeline replaced old alternatives with new ones. This caused cascading failures: a prompt rollback would trigger regressions against the "new" ground truth that no longer contained the old valid outputs. The append-only model eliminates this entirely.

Single-judge evaluation had 89% agreement with human annotations. Three judges with majority vote reached 94%. But three judges with debate resolution reached 97%. The arbiter's structured review of disagreements caught subtle reasoning errors that voting alone missed.
